## Kontext

MCP servery jsou způsob, jak agenti získávají nástroje – přístup k souborům,
scraping webu, databázové dotazy. Jenže MCP server je *kód, který spouštíte
s oprávněními svého agenta*, a komunitní zvyk je pustit ho rovnou z registru:
`npx -y nejaky-mcp-server`. Nikdo si ho předtím nepřečte. Je to tentýž problém
důvěry, který [agent-audit](https://github.com/ondraulehla/agent-audit)
detekuje v konfiguraci – tady jsem chtěl nebezpečí odstranit, ne jen označit.

## Problém

Škodlivý nebo prostě kompromitovaný MCP server běží se vším, co máte: přečte
`~/.ssh`, váš prohlížečový profil, každé tajemství exportované v shellu a dosáhne
na vaši lokální síť. Statická analýza vás varuje; kód, jakmile jednou běží,
už neuhlídá.

## Řešení

**Transparentní stdio most.** `mcp-sandbox` spustí jednorázovou
[E2B](https://e2b.dev) micro-VM, nastartuje MCP server *uvnitř ní* a přemostí
JSON-RPC oběma směry:

```bash
npx mcp-sandbox run -- npx -y nejaky-mcp-server-z-internetu
```

Pro MCP klienta (Claude Code, Cursor, …) to *je* ten server – protokol se
přenáší doslovně. Server ale běží v cloudu s:

- **žádným přístupem** k vašim souborům, prostředí ani lokální síti
- **přenosem env jen přes allowlist** (`--env KEY` / `--env KEY=value`) – přesný
  opak obvyklého „zděď všechno"
- **`--trace`** zachytávajícím každou JSON-RPC zprávu do JSONL k inspekci
- **TTL**, které sandbox sklidí za všech okolností

Most je line-buffered, takže klient nikdy neuvidí půlku JSON-RPC rámce, a celá
E2B integrace sedí za injektovaným rozhraním `SandboxBackend` – díky tomu sada
11 testů běží offline s in-memory fake implementací.

## Výsledek

- **Ověřeno naživo**: sandboxovaný `@modelcontextprotocol/server-everything`
  dokončil `initialize` handshake a přes most vrátil svých 13 nástrojů, provoz
  zachycen do trace souboru
- Poloměr výbuchu pryč: nepřátelský server ublíží jen své jednorázové VM
- Zapadne do existujícího setupu bez úprav kódu:
  `claude mcp add nazev -- npx -y mcp-sandbox run -- <server>`

## Co jsem se naučil

Zajímavým bugem byl cold-start závod: `npx` stahuje balík při prvním spuštění
a uprostřed startu se re-execuje, takže okamžitý `initialize` od klienta se
ztratil. Diagnostika znamenala sondovat sandbox přímo – ověřit runtime, izolovat
okno nestabilního PID – a oprava (předinstalovat v setup kroku, spouštět hotový
binárník) vzešla přímo z pochopení, *proč* se identita procesu mění. Stavba
proti reálnému cloudovému runtime tenhle druh ladění od základů odměňuje.
