## Kontext

Dokumentace hnije potichu. Nikdo nespouští vlastní quickstart poté, co ho
napsal, takže první rozbitá zkušenost vždycky dopadne na nového uživatele –
člověka s nejmenším kontextem a nejmenší trpělivostí. Unit testy to nezachytí:
testují kód, ne příkazy, které lidem říkáte napsat.

`readme-ci` dělá z README testovaný artefakt.

## Jak to funguje

- **Extrakce** – kódové bloky se z markdownu vytáhnou i s čísly řádků;
  `bash`/`sh` běží jako shellové skripty, u `console` bloků se spustí jen řádky
  s `$`, `js` a `python` jdou přes interpret a nespustitelné bloky (`json`,
  `yaml`, …) se automaticky ignorují.
- **Model sezení** – všechny bloky souboru běží v jednom sandboxovém sezení.
  Pracovní adresář a exportované proměnné se přenášejí z bloku do bloku, takže
  quickstart, který v jednom bloku udělá `cd my-app` a v dalším `npm test`, se
  chová přesně jako u čtenáře.
- **Izolace** – bloky běží v jednorázovém Docker kontejneru, případně v cloud
  sandboxu [E2B](https://e2b.dev), když Docker není po ruce; pro důvěryhodnou
  dokumentaci existuje lokální runner. První rozbitý blok shodí soubor,
  vypíše zachycený výstup a v GitHub Actions anotuje přesný řádek markdownu.
- **Řízení** – výjimky řeší komentáře u bloků: `<!-- readme-ci skip -->`,
  `timeout=N`, `cwd=cesta`, `KEY=hodnota`.

## Technické poznámky

Nástroj je TypeScript balíček bez závislostí – malý parser, plánovač a tři
runner backendy za jedním rozhraním – s vitest sadou a CI pipeline, která
`readme-ci` pouští na jeho vlastní dokumentaci ve všech třech runnerech.
Dogfooding se vyplatil okamžitě: odhalil skutečnou chybu, kdy npm exportuje
proměnnou prostředí s pomlčkou ve jméně, což rozbíjelo obnovu stavu mezi bloky.

Tento web si své README hlídá pomocí `readme-ci` také.
